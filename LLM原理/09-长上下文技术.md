# 九、长上下文技术

## 【43】上下文窗口

**是什么**：模型一次能处理的 token 最大数量。受三重限制：注意力 O(n²) 计算复杂度、位置编码的外推能力、训练数据中长序列的比例。

**为什么这样设计**：上下文窗口不是越大越好——从 4K 升到 32K，注意力矩阵变大 64 倍，训练成本暴涨。而且 RoPE 在超出训练长度时高频分量（近处依赖）和低频分量（远处依赖）的衰减比例失衡，可能同时损害长文本和短文本的表现。所以扩展窗口通常需要"继续训练"阶段——用更长序列再训几千步，让位置编码和注意力适应新的序列长度。100 万 token 窗口（Gemini 1.5）靠的是 Ring Attention（序列分片多设备）、FlashAttention（GPU 显存优化）、位置外推技术（RoPE 缩放）的组合拳。

**和周边的联系**：上下文窗口是 KV Cache 大小的上限。窗口越大 KV Cache 显存越吃紧。位置外推和稀疏注意力是扩展窗口的两条主要技术路线。

### 面试问答

**Q：窗口受哪些因素限制？**

A：注意力 O(n²) 随长度暴涨；位置编码训练时只见过短序列；训练数据里缺少长序列样本。三重约束。

**Q：扩大窗口对性能的影响？**

A：训练成本暴涨（32K 约是 4K 的 8 倍）；可能同时损害长文本和短文本性能；通常需继续训练适应。

**Q：100 万 token 窗口怎么做到？**

A：Ring Attention（序列分片）、FlashAttention（显存优化）、RoPE + NTK 缩放（位置外推）、多机共享 KV Cache。组合拳。

## 【44】位置外推

**是什么**：让模型能处理比训练时更长的序列。核心做法是调整 RoPE 的位置索引——把新的"大位置"映射到训练见过的"小位置"范围内。

**为什么这样设计**：RoPE 天然利于外推，因为它编码的是相对距离而非绝对位置。训练时未见过的只有那些超大距离（如位置 1 和位置 32768 之间的差距，如果训练最大 4096），大部分短距离在训练中已经充分见过。线性缩放最简单粗暴（所有位置索引乘 0.25 → 训练 4K 映射到推理 16K），但会压坏短距离精度。NTK 缩放更聪明——RoPE 使用多个频率（高频负责近处、低频负责远处），NTK 只调低频（让远处能看更远），不动高频（保持近处分辨率）。NTK 基本不需要微调，效果还好。

**和周边的联系**：位置外推和位置编码（RoPE）直接相关。它是扩展上下文窗口的核心技术，比重训练更经济。

### 面试问答

**Q：为什么 RoPE 利于外推？**

A：RoPE 编码相对距离。训练时已见过绝大多数短距离，只需把新的远距离映射到已见范围。

**Q：线性缩放和 NTK 的区别？**

A：线性缩放把所有位置索引用一个系数压缩，简单但伤短距离精度。NTK 智能缩放——只调低频分量（远距离），高频不动（近距离）。效果更好，基本无需重训。

**Q：需要微调吗？**

A：线性缩放通常需要（几百到几千步）。NTK 零微调友好，但要最佳效果仍建议少量继续训练。

## 【45】稀疏注意力

**是什么**：不计算所有 token 两两之间的注意力（O(n²)），只算一部分——比如每个 token 只看附近 ±w 个（滑动窗口），或隔 d 个看一个（步长采样），或设少数"哨兵 token"跟所有人交互（全局注意力）。

**为什么这样设计**：全注意力的 O(n²) 对于长序列是不可承受的——10 万 token 的注意力矩阵有 100 亿个元素。稀疏注意力的直觉是：大多数 token 之间的依赖都很弱，不需要两两都算。滑动窗口保证局部连贯性（词和它的邻居最相关），步长采样提供远处概览（每隔一段看一眼），全局哨兵充当信息枢纽（重要 token 负责传递信息）。经过多层堆叠，稀疏窗口的信息也能传播到远处（第 N 层有效感受野 ≈ N × 窗口大小）。精度损失通常 1%~3%。不过近年 FlashAttention 等全注意力优化技术做得越来越好，稀疏注意力的优势在缩小。

**和周边的联系**：稀疏注意力和 FlashAttention 是解决注意力长序列瓶颈的两条路径——前者减少计算，后者优化计算。和位置外推互补：外推解决了位置编码的兼容性问题，稀疏解决计算成本问题。

### 面试问答

**Q：典型稀疏模式？**

A：滑动窗口（只看附近 ±w）、步长（隔 d 个跳读）、全局哨兵（少数 token 和所有人交互）。通常组合使用。

**Q：精度损失大吗？**

A：很小，1%~3%。靠信息在多层间传播——N 层 × 窗口大小 = 有效感受野。全局哨兵充当前信息枢纽。

**Q：著名模型？**

A：Longformer、BigBird、Sparse Transformer。近年趋势 FlashAttention 优化全注意力更主流。

## 【46】RAG（检索增强生成）

**是什么**：先拿用户问题去外部知识库检索相关文档，把检索到的文档块和问题一起塞进模型上下文，模型基于这些"参考资料"生成答案。

**为什么这样设计**：模型内部知识有两个致命缺陷——不更新（训练日截止后的事全不知道）和不精准（对具体细节容易记混）。RAG 的解法是把生成从"闭卷考试"变成"开卷考试"：回答前先翻书查资料，基于查到的权威内容来回答。检索部分一般用嵌入模型（把文档和问题都编码成向量，找最相似的）+ BM25（关键词精确匹配）做混合检索。RAG 能把幻觉率从 20%~30% 降到个位数，而且答案可溯源——知道是从哪篇文档来的。适合知识频繁更新、需要明确引用来源的企业场景。

**和周边的联系**：RAG 和 Function Calling 是减轻幻觉的两大手段。RAG 和长上下文窗口互补——RAG 选相关文档后交给长窗口做综合分析。技术栈涉及嵌入模型、向量数据库、混合检索策略。

### 面试问答

**Q：检索部分用什么模型？**

A：传统 BM25（关键词，不用训），现代密集检索（嵌入模型，语义匹配）。最佳实践是混合检索——BM25 保证精准 + 密集保证语义。

**Q：RAG 怎么减少幻觉？**

A：闭卷变开卷。不靠内部记忆，先查资料再基于资料回答。检索靠谱的话幻觉率从 20~30% 降到个位数。

**Q：RAG 和长上下文窗口哪个好？**

A：互补。RAG 适合知识频繁更新、需出处。长窗口适合深度理解整个文档。可结合——RAG 检索再用长窗口分析。
