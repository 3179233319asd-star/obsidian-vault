# 八、工程优化与大规模训练

## 【39】混合精度训练

**是什么**：用 16 位浮点数（FP16 或 BF16）做前向和反向计算，同时保留一份 FP32 的主权重副本用于精确的参数更新。这样既有 16 位的速度，又有 32 位的数值稳定性。

**为什么这样设计**：32 位浮点数在很多场景下是"过度精确"——训练 LLM 不需要小数点后七八位的精度。切成 16 位后，每个参数占的显存减半，而且现代 GPU（A100/H100）对 16 位运算有专门的硬件加速，速度能翻倍。BF16 比 FP16 更适合 LLM——BF16 的数值范围跟 FP32 一样大（因为指数位数相同），所以梯度基本不会溢出；FP16 范围小得多，需要额外做损失缩放来防止小梯度变成零。实际显存省约 30%~40%（因为 FP32 主权重和优化器状态仍需部分 32 位存储）。

**和周边的联系**：混合精度是其他大规模训练技术（梯度累积、ZeRO）的基础。几乎所有现代 LLM 训练都默认用 BF16 + 混合精度。

### 面试问答

**Q：FP16 和 BF16 的区别？**

A：FP16 范围小易溢出，需损失缩放。BF16 范围跟 FP32 一样大，梯度不易溢出，A100/H100 有硬件加速。LLM 首选 BF16。

**Q：损失缩放是什么？**

A：FP16 梯度太小会变 0（下溢）。前向时 loss 乘大因子 → 梯度放大 → 避免下溢 → 更新前除回来。BF16 不需要。

**Q：能省多少显存？**

A：理论一半，实际 30%~40%。参数和梯度变 16bit，但 FP32 主权重和优化器状态仍需部分 32bit。

## 【40】梯度累积

**是什么**：不每算一个小 batch 就更新参数，而是连续算 N 个小 batch，把梯度累加到一个累加器里，攒够了 N 步才用总梯度更新一次参数。

**为什么这样设计**：大 batch 训练更稳定，但每张 GPU 的显存放不下那么多样本。梯度累积用时间换空间——每个小 batch 只装显存能塞得下的样本数，算完梯度加进累加器，相当于在模拟"把 N 个小 batch 拼成一个大 batch"。数学上完全等价（梯度是线性的，累加 = 大 batch 的梯度）。LLM 用 LayerNorm 不受 BatchNorm 的小 batch 统计偏差影响，所以效果完全一致。唯一的代价是训练速度变慢（每个小 batch 都要跑一次前向+反向）。

**和周边的联系**：梯度累积通常和混合精度、ZeRO 组合使用。步数 = 目标全局 batch ÷ 单卡 batch ÷ GPU 数。

### 面试问答

**Q：怎么实现大 batch？**

A：多个小 batch 算梯度不更新 → 累加 → 攒够 N 步 → 用总梯度更新。等效 batch size = 单步 N 倍。时间换空间。

**Q：步数怎么设？影响效果吗？**

A：步数 = 目标 batch ÷ 单卡 batch ÷ GPU 数。数学上完全等价（梯度是线性的），LLM 用 LayerNorm，实践中无差异。

## 【41】ZeRO / FSDP

**是什么**：把模型的参数、梯度、优化器状态切分成 N 份，分别存到 N 张 GPU 上，每张 GPU 只存 1/N，需要时再从其他 GPU 临时收集完整数据。

**为什么这样设计**：一张 GPU 的显存放不下几百亿参数的完整模型、梯度和优化器状态（全部加起来可能超过 2TB）。ZeRO 的核心思路是"既然装不下，就拆开存"——优化器状态是最大头（AdamW 存两个动量 + FP32 主权重，约 12× 参数量），所以 ZeRO-1 先分片这个。还不够就再分片梯度（ZeRO-2），再不够连参数也分片（ZeRO-3）。阶段越高，通信越多（每次前向都要从其他 GPU 收集参数），但在高速互联（NVLink）下，可接近线性扩展。FSDP 是 PyTorch 官方的 ZeRO-3 实现。

**和周边的联系**：ZeRO/FSDP 结合混合精度和梯度累积，是目前训练千亿级模型的标配组合。ZeRO-3 通信开销最大但能训的模型也最大。

### 面试问答

**Q：ZeRO 的三个阶段？**

A：ZeRO-1 分片优化器状态（大头，占约 12× 参数），ZeRO-2 额外分片梯度，ZeRO-3 参数也分片。阶段越高通信越多。

**Q：FSDP 和 ZeRO-3 的异同？**

A：FSDP 是 PyTorch 官方的 ZeRO-3 实现，按层分片，支持参数 offload 到 CPU。核心机制类似。

**Q：通信开销？**

A：ZeRO-3 每次前向/反向前要 all-gather 收集完整参数。配合高速互联和梯度累积，可接近线性扩展。

## 【42】MoE（混合专家模型）

**是什么**：把 Transformer 每一层的 FFN 替换成多个"专家"（每个专家就是一个小 FFN），由门控网络为每个 token 选 1~2 个最合适的专家来处理。总参数量大增，但每个 token 实际只激活了其中一小部分。

**为什么这样设计**：传统稠密模型的问题是——参数量和计算量绑定在一起，增加参数就增加计算。MoE 做了一个"解耦"：参数存很多（比如 8 个专家），但每个 token 只用 2 个，所以计算量 ≈ 2/8 的稠密模型。这就像一个大公司有 8 个专家，每个问题只用最擅长的那几个来回答，不用全员出动。代价是显存仍需装下所有专家，而且门控如果不做负载平衡，热门专家会被挤爆、冷门闲置。Mixtral 8×7B（总参 47B，实际计算约 13B）是目前最知名的开源 MoE 模型。

**和周边的联系**：MoE 是 FFN 的扩展。门控机制和注意力层的选择有本质区别——注意力是 token 之间的交互，门控是 token 对专家的路由。MoE 通常配合 ZeRO 等并行策略一起训练。

### 面试问答

**Q：怎么平衡专家负载？**

A：门控为每个 token 选 Top-2 专家。用负载平衡辅助损失惩罚偏科行为，鼓励均匀分配。MoE 省的是计算量，不是显存。

**Q：为什么推理比同参数量稠密模型快？**

A：总参数多但每步只激活少数专家。Mixtral 47B 总参数，实际计算量约 13B。参数量和计算量解耦。

**Q：知名模型？**

A：Switch Transformer、GShard、Mixtral 8×7B（最知名开源）、DeepSeek-V2、Qwen2-MoE。
