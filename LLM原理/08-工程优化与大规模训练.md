# 八、工程优化与大规模训练

## 【39】混合精度训练

### 用人话解释

训练模型用的数字默认是 32 位浮点数（FP32），精确到小数点后六七位。但实际上训练 LLM 不需要那么精确——16 位就够用了。切成 16 位后每个参数占的内存减半，而且现代 GPU（A100/H100）对 16 位运算有硬件加速，速度直接翻倍。

但 16 位数字有个问题：数值范围变小了。FP16 的范围小（最大约 65504），梯度容易溢出。所以 LLM 训练用 BF16——范围跟 FP32 一样大，只是精度低一点点。BF16 训练几乎不会溢出，FP16 还需要额外做"损失缩放"来防止小梯度变成零。

> 你可以这样记：FP32 = 用高精度计算器算账，BF16/FP16 = 用普通精度算，够用了还快两倍省一半内存。LLM 训练全用 BF16。

### 面试怎么说

**Q：FP16 和 BF16 区别？**

A：FP16 范围小易溢出，需损失缩放。BF16 范围跟 FP32 一样大，梯度不溢出且 GPU 有硬件加速。LLM 首选 BF16。

**Q：能省多少？**

A：理论一半显存，实际 30%~40%。参数和梯度变 16bit，但 FP32 主权重和优化器状态仍需部分 32bit。

## 【40】梯度累积

### 用人话解释

理想情况下 batch 越大训练越稳定，但每张 GPU 显存放不下那么多样本。梯度累积的解法：跑 N 个小 batch，每小批算完梯度不马上更新参数，而是把梯度加进累加器。攒够 N 次后，累加器里的总梯度就等于一个大 batch 的梯度，此时一次更新参数。用时间换空间。

数学上完全等价——因为梯度就是损失对参数的导数，多个小 batch 的梯度加起来就等于大 batch 的梯度。LLM 用 LayerNorm 不受 BatchNorm 的小 batch 统计量偏差影响。

> 你可以这样记：梯度累积 = 攒零钱。每次只有几块钱不够买东西，攒到月底一次花。每笔小钱都记着账，攒出来的总额跟一开始就有一大笔钱完全等价。

### 面试怎么说

**Q：怎么实现大 batch？**

A：多小 batch 算梯度不更新 → 累加 → 攒够 N 步 → 一次性更新。等效 batch size = 单步 × N。优化效果完全等价（梯度是线性的）。

**Q：步数怎么设？**

A：步数 = 目标 batch ÷ 单卡 batch ÷ GPU 数。太多步训练变慢。LLM 用 LayerNorm 无统计偏差问题。

## 【41】ZeRO / FSDP

### 用人话解释

训练千亿参数模型时，单张 GPU 连模型权重都放不下，更别说梯度、优化器状态了。ZeRO 的思路：既然一张卡放不下，就把这些东西切碎分到多张卡上，每张只存一小块，需要用到完整参数时再从其他卡临时收集。

分三个阶段——ZeRO-1 先分片优化器状态（占显存最大头，约参数量的 12 倍）；ZeRO-2 额外分片梯度；ZeRO-3 连模型参数也分片了。阶段越高能训练越大的模型，但通信开销也越大（每次前向/反向前要从其他 GPU 收集参数）。FSDP 是 PyTorch 官方的 ZeRO-3 实现。

> 你可以这样记：ZeRO = 一句话太长一个人记不住，分成几段每人记一段，需要的时候互相问。阶段 1~3 分的越来越细，能训的模型越来越大，但互相问的次数也越来越多。

### 面试怎么说

**Q：三个阶段？**

A：ZeRO-1 分片优化器状态（最大头），ZeRO-2 加梯度分片，ZeRO-3 参数也分片。阶段越高通信越多。

**Q：FSDP 是什么？**

A：PyTorch 官方的 ZeRO-3 实现，按层分片，支持 offload 到 CPU。核心机制跟 ZeRO-3 相同。

## 【42】MoE（混合专家模型）

### 用人话解释

传统模型的问题是参数量和计算量绑在一起。MoE 的鬼才思路：把 FFN 换成 8 个"专家"（每个专家就是一个独立的 FFN），每次处理一个 token 时门控网络只挑 1~2 个最合适的专家去处理。结果是：总参数量大增（8 个专家的总和），但每个 token 的计算量只增加了一点点（只激活 2 个）。

就像一个公司有 8 个专业顾问，但每次客户来电只派最擅长该领域的 2 个人去处理——公司的"总咨询能力"很大，"每次处理成本"却不高。Mixtral 8×7B 就是最知名的开源 MoE：47B 总参数，但实际计算量约等于 13B 稠密模型。

> 你可以这样记：MoE = 不养一个全能员工（太贵训不出来），养 8 个专才，每次只让 2 个干活。实力强但开销不涨太多。

### 面试怎么说

**Q：怎么平衡专家负载？**

A：门控选 Top-2，用负载平衡辅助损失惩罚偏科行为——别全挤给一两个热门专家。MoE 省的是计算量不是显存。

**Q：为什么推理比同参数稠密模型快？**

A：总参数大但每次只激活少数专家。Mixtral 47B 总参数 ≈ 13B 的实际计算量。参数量和计算量解耦。

**Q：知名模型？**

A：Switch Transformer、GShard、Mixtral 8×7B、DeepSeek-V2、Qwen2-MoE。
