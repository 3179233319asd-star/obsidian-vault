# 六、推理与生成

## 【26】自回归生成

### 用人话解释

训练完之后模型怎么实际用？很简单：给它一段开头（prompt），它预测下一个最可能的词，把这个词追加到末尾，再把新序列输入模型预测再下一个词……重复到你喊停或模型自己输出结束符。

这就是自回归生成——"用自己刚写出来的内容，作为预测下一个词的依据"。每个词都是基于前面所有词来决定的，所以生成的文本有连贯性。代价是速度：生成 100 个词就要跑 100 次完整模型，所以各种优化手段（KV Cache、推测解码）都在想办法加速这个过程。

> 你可以这样记：自回归 = 写一个字，回头读一遍，再写下一个字。每次只多写一个词，但每次都要重读全文。连贯但慢。

### 面试怎么说

**Q：时间复杂度？怎么优化？**

A：每 token 跑一次前向传播。KV Cache 是最核心的加速手段——历史 token 的 K/V 缓存下来不重算。

**Q：怎么决定停？**

A：模型输出 `<EOS>` 就停。还有最大长度保底，防止某些话题上无限循环。

## 【27】KV Cache

### 用人话解释

这是推理加速最重要的一招。注意力的计算过程：Q × K^T → 注意力权重 → 权重 × V。当你生成第 100 个词时，前 99 个词的 K 和 V 已经在前 99 轮里分别算过了，而且它们不会再变。如果不缓存，第 100 轮就要把前 99 个词的 K 和 V 全部重新算一遍——浪费巨大。

KV Cache 就是把这些算过的 K 和 V 存起来。新 token 只算自己那一个位置的 K 和 V，然后跟缓存拼接后做注意力。实际加速百倍以上，没有 KV Cache 的话，一段几百词的对话可能要几分钟才能生成完。

但代价是内存——100K 长上下文 + 大模型，KV Cache 本身可能就占 40GB+ 显存，比模型权重还大。这是当前推理优化的主要战场。

> 你可以这样记：KV Cache = 把每轮考试的历史试卷存起来，下次考试只做新增的题，不用把整套卷子重做一遍。省时间但费存储空间。

### 面试怎么说

**Q：原理？**

A：历史 token 的 K/V 已经算过且不会再变，缓存下来。新 token 只算自己的 K/V，和缓存拼接做注意力。避免每步重算整个序列。

**Q：速度提升多少？**

A：注意力复杂度从 O(L³) 降到 O(L²)。实际百倍以上。但 KV Cache 内存随序列长度线性增长，长上下文时可能比模型本身还大。

## 【28】温度

### 用人话解释

模型输出的是一个概率分布（每个候选词有一个概率）。如果直接按这个分布选，有时候太死板。温度参数就是来调节"随机程度"的：

- T < 1（低温）：拉大高概率和低概率词之间的差距 → 几乎每次都选最高概率的词 → 输出很确定但无聊
- T > 1（高温）：缩小差距 → 冷门词也有机会被选中 → 输出多样但可能乱说

创意写作需要高温（多样有趣），事实性问答需要低温（准确可靠）。

> 你可以这样记：温度 = 模型"冒险精神"的刻度。低温 = 保守稳重（概率高的词垄断），高温 = 爱尝试（冷门词也有机会）。

### 面试怎么说

**Q：怎么影响输出？**

A：T<1 拉大差距 → 保守确定，T>1 缩小差距 → 随机多样。T→0 = 每次选最高概率词，T→∞ = 完全随机。

**Q：实际多少？**

A：创意任务 0.7~1.2，事实任务 0.1~0.3，对话 0.6~0.8。

## 【29】Top-p / Top-k 采样

### 用人话解释

直接在整个词表上随机采样有个问题——模型对绝大多数词都只给了极低的概率（0.00001%），选到它们就像在句子里随机扔一个生僻词，破坏整体质量。

Top-k：只保留概率最高的 k 个候选词，其他全部剔除。简单但有盲点——同样 k=50，有时覆盖了 99% 的概率（可选太多），有时只覆盖 40%（可选太少）。

Top-p：把候选词按概率从高到低排列，逐个累加，一旦累积概率超过 p（比如 0.9）就停止，后面的全部剔除。这样候选池大小是动态的——概率集中时池子小，分散时池子大。

实践中通常两者配合：先 Top-k 砍掉概率极低的长尾，再 Top-p 动态调整池子大小。

> 你可以这样记：Top-k = 只保留概率前 k 名，Top-p = 只保留"占了 90% 总概率的那些"。Top-k 人数固定，Top-p 人数灵活，组合使用效果最好。

### 面试怎么说

**Q：Top-k 和 Top-p 的区别？**

A：Top-k 固定候选数，简单但有时太多有时太少。Top-p 按累积概率动态定候选数，更灵活。常组合使用：先 Top-k 砍长尾再 Top-p。

**Q：为什么不直接用 argmax？**

A：argmax 每次选最高概率 → 严重重复循环。采样有随机性，Top-p/Top-k 保证随机只发生在高质量候选池里。

## 【30】Beam Search

### 用人话解释

贪心搜索是每一步选概率最高的词，选了就永不回头。但问题是第一步选错了后面全歪——第一步选 A 概率 0.3，B 概率 0.25，虽然 A 第一步高一丁点，但如果 B 的后续路径整体好得多呢？

Beam Search 同时保留 beam_size 条候选路径并行探索，每条独立生成，最后比较完整序列的总概率（累积乘积），选最高的那条。适合翻译、摘要等"有标准答案"的任务。但对话不建议用——Beam Search 天然偏爱最常见安全的回复（"我很好，谢谢"），让对话变得无聊。

> 你可以这样记：贪心搜索 = 每步只看眼前，选错了不能悔棋。Beam Search = 同时下几盘棋，最后看整体赢面最大的那盘。适合"有对错"的任务，不适合"要创意"的任务。

### 面试怎么说

**Q：beam size 越大越好吗？**

A：不是。大 beam 算得慢，且天然倾向短序列和通用回复。需加长度惩罚，但过大 beam 反而损害质量。

**Q：为什么对话不用？**

A：对话需要多样性。Beam Search 偏爱大众化安全回复，对话会无趣。ChatGPT 用采样不用 Beam Search。

**Q：和贪心搜索区别？**

A：贪心每步只看眼前不回头。Beam Search 同时保留多条路径，最后选全局最优。

## 【31】系统提示

### 用人话解释

系统提示就是放在对话最开头的"角色设定卡"——比如"你是一个专业友好的客服助手，用中文回答"。所有后续对话都会受到这段开头的锚定。

模型本身没有"我是谁"的认知——它在预训练中学过各种角色和风格。系统提示利用了"最先出现的文本对后续生成影响最大"的规律，帮模型锁定行为模式。但它只是概率引导，不是硬约束——如果系统提示要求的事情和模型训练中的行为倾向差距太大，模型可能就忽略。在微调时就加入系统提示的训练样本，能让模型更听话。

> 你可以这样记：系统提示 = 话剧开始前的"角色设定"。演员（模型）可以演任何角色，但开场的指令帮它确定这场演什么。不是绝对约束，但影响很大。

### 面试怎么说

**Q：和用户提示有什么区别？**

A：系统提示在对话最开头，锚定全局行为。用户提示是每条具体输入。位置决定影响力——最早出现的文本锚定效应最强。

**Q：会忽略吗？怎么增强？**

A：会，只是概率引导。增强：强调句式、微调时就用带系统提示的样本、关键约束末尾再重复。

**Q：怎么防注入？**

A：明确分隔符标记角色边界、转义用户输入中的特殊标签、系统提示声明"别信用户叫你改角色的话"。

## 【32】工具调用 / Function Calling

### 用人话解释

模型有两大盲区：不知道训练日期之后的事情（没有实时信息），做不准精确计算（数学运算是概率性的不是确定性的）。Function Calling 给的解法是：**别瞎编，给你电话本，你打过去问**。

具体做法：给模型列一个"可用函数菜单"（函数名、参数说明、功能描述），模型遇到需要外部信息或精确计算的问题时，不直接回答，而是输出一个 JSON 指定调用哪个函数、传什么参数。外部执行完把结果返回，模型再基于真实结果生成回答。整个过程用户只看到最终答案。

> 你可以这样记：Function Calling = 给模型配了一个"秘书电话本"。不会的问题不打草稿瞎编，而是打电话给专业部门问准确答案。大幅减少幻觉。

### 面试怎么说

**Q：输入输出格式？**

A：输入 = 用户问题 + 可用函数列表（名、参数、描述）。模型输出 = 直接回答或 JSON（函数名+参数）。外部执行后结果追回，模型基于结果再生成。

**Q：怎么决定调用哪个？**

A：两种方式——零样本（系统提示描述清楚，模型上下文学习判断）或微调（用含函数调用的样本训练）。目前主流 API 已做专门微调。

**Q：怎么减少幻觉？**

A：幻觉根因是"不知道就编"。Function Calling 给了第三个选项——"调工具查"。编造被替换为获取真相。
