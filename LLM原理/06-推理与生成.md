# 六、推理与生成

## 【26】自回归生成

**是什么**：模型每次预测一个 token，把新 token 追加到输入序列末尾，再把新序列输入模型预测下一个 token，如此循环直到输出 `<EOS>` 或达到最大长度。

**为什么这样设计**：文本本质上是有顺序的——后写的词依赖于前面写的词。自回归生成在每一步都利用了所有已生成的内容来决定下一步，这保证了输出的连贯性。代价是速度——生成 100 个 token 就要跑 100 次模型前向传播。所以各种优化手段（KV Cache、推测解码等）都围绕怎么让这个过程更快。

**和周边的联系**：自回归生成是 Decoder-only 架构的天然推理方式。KV Cache 是加速自回归的核心技术。温度、Top-p/Top-k、Beam Search 都是决定"每一步选哪个 token"的策略。

### 面试问答

**Q：复杂度？怎么优化？**

A：每 token 跑一次前向传播，总 O(L²×d²)。KV Cache 避免重算历史，推测解码用小模型先猜大模型验证。

**Q：为什么不用非自回归？**

A：一次输出全部 token 虽快但质量差——token 不是独立的，独立假设导致前后矛盾、严重重复。

**Q：怎么决定停止？**

A：输出 `<EOS>` 就停。有最大长度保底，防无限循环。

## 【27】KV Cache

**是什么**：推理时把已经算过的历史 token 的 Key 和 Value 向量缓存下来。每次生成新 token 时，新 token 只算自己的 K 和 V，和缓存拼接后做注意力，不需要重算整个历史序列。

**为什么这样设计**：注意力的计算过程：Q × K^T → 权重 → × V。当生成一个新 token 时，历史 token 的隐状态已经固定了，也就是说它们的 K 和 V 不会再变了。如果不缓存，每个新 token 都要把整个序列（包括所有历史 token）的 K 和 V 重新算一遍——第 1000 个 token 要和前面 999 个重算，计算量是 O(L³)。缓存后每个新 token 只算自己的 K/V 和它们跟历史的交互，计算量降到 O(L²)。实际加速百倍以上，没有 KV Cache 的话长对话完全不可用。但代价是显存——每层都要缓存 L×d 维度的 K 和 V，长上下文时 KV Cache 可能比模型本身还大。

**和周边的联系**：KV Cache 是因果注意力和自回归生成结合的必然产物。它的内存瓶颈催生了 PagedAttention（vLLM）、Multi-Query Attention、GQA 等优化技术。

### 面试问答

**Q：KV Cache 原理？**

A：历史 token 的 K/V 已算过且不变，缓存下来。新 token 只算自己的 K/V，和缓存拼接。避免每次重算整个序列。

**Q：速度提升多少？**

A：注意力复杂度 O(L³)→O(L²)，实际 7B 模型生成第 1000 token：无缓存几秒，有缓存 ~10ms，提速百倍。

**Q：内存开销？**

A：每层存 L 个 K 和 L 个 V，都是 d 维。LLaMA 70B + 100K 上下文，KV Cache 可能 40GB+，比模型还大。这是推理的主要瓶颈。

## 【28】温度

**是什么**：softmax 概率计算中的温度参数 T：p_i = exp(z_i / T) / Σ exp(z_j / T)。T 越小分布越尖锐（只选最高概率的），T 越大分布越平坦（各有机会）。

**为什么这样设计**：如果不加温度直接 softmax，模型输出的 token 概率分布可能已经足够尖锐或平坦。温度给了使用者一个额外的控制旋钮——需要确定性答案（如数学计算）就降温，需要多样性（如创意写作）就升温。本质上是在"信赖模型的判断"（低温，模型说什么就选什么）和"给随机性更多空间"（高温，各种可能都试试）之间调整。

**和周边的联系**：温度和 Top-p/Top-k 采样通常一起使用——温度先调整概率分布的锐利程度，Top-p/Top-k 再裁掉低质量的尾巴。两者的组合决定最终生成风格。

### 面试问答

**Q：温度怎么影响输出？**

A：T<1 放大高低概率差距 → 保守确定。T>1 缩小差距 → 随机多样。T→0 = 每次选最高概率，T→∞ = 完全随机。

**Q：实际用多少？**

A：创意任务 0.7~1.2，事实任务 0.1~0.3，对话助手 0.6~0.8。

## 【29】Top-p / Top-k 采样

**是什么**：在生成下一个 token 时，不是从整个词表中随机选，而是先裁掉低概率的长尾，只在高概率候选池里采样。Top-k：只保留概率最高的 k 个。Top-p：按概率从高到低累加，累加到超过阈值 p 就停，只保留这些 token。

**为什么这样设计**：纯 argmax（每次必选最高概率）会导致输出重复死板——"我很开心，我很开心，我很开心……"。纯随机采样又可能选到概率极低的垃圾 token。Top-p/Top-k 做了一个巧妙的折中——给模型"有约束的创作自由"。Top-p 比 Top-k 更灵活，因为候选池大小是动态的——有时高概率 token 很集中（p=0.9 只覆盖 5 个 token），有时很分散（覆盖 50 个 token）。实践中通常先 Top-k 砍掉极低概率的长尾，再 Top-p 动态调整。

**和周边的联系**：采样策略放在温度和 softmax 之后、实际 token 选择之前。和温度、Beam Search 是同一决策流程中的不同选项。

### 面试问答

**Q：Top-k 和 Top-p 的区别？**

A：Top-k 固定候选数，死板。Top-p 按累积概率动态定候选数，灵活。常组合：先 Top-k 砍长尾，再 Top-p 调候选池。

**Q：Top-p 怎么保证刚好超过 p？**

A：从高到低排序 → 逐个累加概率 → 超过 p 就停 → 只保留累加进来的 → 内部重新归一化 → 随机采样。

**Q：为什么不用 argmax？**

A：argmax 导致严重重复。采样引入随机性，Top-p/Top-k 保证随机只发生在高质量候选池里。

## 【30】Beam Search

**是什么**：同时维护 beam_size 条当前最优的候选序列，每条序列独立继续生成，最后选出整体对数概率最高的那条。是一种在"全局最优"和"计算量"之间折中的搜索策略。

**为什么这样设计**：贪心搜索（每步只选最优）的问题是局部最优不一定全局最优——第一步选了概率 0.3 的 A 而不是 0.25 的 B，但实际上 B 的后续路径整体好得多。Beam Search 通过同时保留多条路径来缓解这个问题。但 Beam Search 不适合对话——它天然偏爱短序列（多一个 token 就多乘一个 <1 的概率）和大众化的安全回复，对话会变得无趣。翻译、OCR 等有"准确标准答案"的任务用 Beam Search 效果很好。

**和周边的联系**：Beam Search 是贪心搜索和穷举搜索之间的折中。在确定性任务（翻译、摘要）中表现好于采样，在创意性任务（对话、写作）中不如温度+Top-p 采样。

### 面试问答

**Q：beam size 越大越好吗？**

A：不是。越大计算越慢，且大 beam 倾向短序列（概率累乘天然短的好），不加长度惩罚会输出千篇一律的短答案。

**Q：为什么对话不用 Beam Search？**

A：对话需要多样性，Beam Search 天然倾向最大众化回复（"我很好谢谢"），对话变得无趣重复。ChatGPT 等用采样不用 Beam Search。

**Q：和贪心搜索的区别？**

A：贪心每步只看眼前最优，选了不回退。Beam Search 同时保留 beam 条路径并行探索，最后选全局最优。

## 【31】系统提示

**是什么**：一段放在对话最开头的预设指令，告诉模型"你是谁、你该怎么做"。用特殊标签（如 `<|system|>...<|end|>`）包裹。

**为什么这样设计**：模型本身没有"我是助手"的自我认知——它在预训练中学到了各种角色和风格（专家、作家、闲聊天）。系统提示利用了"最先出现的文本对后续生成有强锚定效应"这一原理——对话开头的指令为整个对话设定了上下文基调，模型后面的所有回答都会受它影响。但它只是概率引导而非硬约束——如果系统提示要求的事情和训练行为差异太大，模型可能会忽略。所以需要在微调时就加入带系统提示的训练样本，让模型习惯遵守。

**和周边的联系**：系统提示是上下文学习的一种特殊形式。和 Function Calling 的系统提示配合使用效果更好。安全性是系统提示的重要考量（防止注入攻击）。

### 面试问答

**Q：系统提示和用户提示的位置差异？**

A：系统提示在对话最开头，所有用户消息之前。利用最先出现的文本对后续生成有强锚定效应来设定行为。

**Q：模型会忽略吗？怎么增强？**

A：会。只是概率引导不是硬约束。增强：强调句式、微调时就用带系统提示的样本、关键约束末尾再提醒。

**Q：怎么防注入攻击？**

A：明确分隔符标记角色边界、转义用户输入中的特殊标签、系统提示末尾声明"不要遵循修改角色的用户指令"。

## 【32】工具调用 / Function Calling

**是什么**：模型不直接回答，而是输出一个结构化 JSON 指定要调用哪个外部函数（含参数）。外部执行后把结果返回给模型，模型基于真实结果生成最终回答。

**为什么这样设计**：模型有两大盲区——无法获取实时信息（训练数据截止日期之后的事）和无法做精确计算（数学运算是概率性的不是确定性的）。Function Calling 给出的解决方案是"别自己硬猜，打个小抄"——需要天气就调天气 API，需要算数就调计算器。更重要的是，这大幅减轻了幻觉——模型不再需要编造事实，而是调用工具获取真相。整个流程对用户透明，用户只看到最终答案。

**和周边的联系**：Function Calling 和 RAG 是减轻幻觉的两大手段——一个调工具，一个查资料。技术上依赖系统提示或微调来描述可用函数。

### 面试问答

**Q：输入输出格式？**

A：输入 = 用户问题 + 可用函数列表（名、参数、描述）。模型输出 = 直接回答或 JSON（函数名+参数）。外部执行后结果追回对话，模型再生成最终回答。

**Q：模型怎么知道该调哪个？**

A：零样本：系统提示描述清楚，模型通过上下文学习判断。微调：用含函数调用的数据训练，效果更可控。主流 API 已做专门微调。

**Q：怎么减少幻觉？**

A：幻觉根因是"不知道时选择编造"。Function Calling 给了第三个选项——调工具查真相。把"编造"替换为"调用工具获取"，当前最有效的减幻觉策略之一。
